# Getting Usable Local Elixir Code Out of a 4GB Pascal GPU: The Python-Pivot Idea vs. the Full Toolbox

## TL;DR
- **The Python-pivot idea is real and academically validated as a *training-data* and *test-time* strategy, but NOT as a naive deterministic "transpile Python→Elixir at inference" pipeline** — no working deterministic Python→Elixir transpiler exists, and routing through Python fixes *syntax/idiom* far more than it fixes *logic*. Your OHLC/time-window bugs are largely language-independent reasoning failures that a pivot will only partially rescue (indirectly, by raising model competence, not by making logic "language-dependent").
- **The single highest-leverage, evidence-backed move is the MultiPL-T pattern: use a bigger model once, offline, to generate a synthetic Elixir fine-tuning corpus validated by translated unit tests, then QLoRA-fine-tune a small (1.5B–3B) Qwen2.5-Coder to run on Pascal.** MultiPL-T showed that a *smaller* generator (StarCoderBase-15B) can produce data that improves even much larger models — so your 16–20B Qwen is more than enough to bootstrap a small Elixir specialist. This attacks the "little Elixir data" root cause rather than patching symptoms.
- **A 4GB Pascal card can run a Q4 1.5B–3B code model with llama.cpp (CUDA works; Flash Attention does NOT on Pascal), but expecting a raw 4GB model to one-shot a correct stateful GenServer is "daydreaming."** The realistic architecture is: small fine-tuned Elixir model + grammar-constrained decoding for syntax + an execution/test-driven repair loop (your Credence) for logic — each component is independently evidence-supported.

## Key Findings

**On Pascal hardware (verified against NVIDIA + llama.cpp primary sources):**
- Pascal consumer cards (GTX 10-series, GP104/106) are CUDA compute capability **6.1**. Per NVIDIA's Pascal Tuning Guide, on GP104 "FP16 throughput is lower, 1/64th that of FP32," but it "provides additional high-throughput INT8 support not available in GP100" via the `__dp4a` 4-way 8-bit integer dot-product instruction. There are **no tensor cores** and **no bfloat16** (bf16 first appears on Ampere/A100).
- llama.cpp runs on Pascal via CUDA (confirmed: GTX 1060 6GB, compute 6.1, in the official CUDA benchmark discussion). **Flash Attention does not work on Pascal**: llama.cpp errors with "CUDA kernel flash_attn_ext_f16 has no device code compatible with CUDA arch 610" and falls back to a slow vec kernel; "Pascal GPUs (CC 6.1) lack MMA and WMMA instructions." So the practical stack is GGUF + llama.cpp CUDA, FA off, INT-quantized weights (which align with Pascal's dp4a INT8 strength rather than its crippled FP16).
- Quantization reality for small models: practitioner and academic sources converge — "a 3B at Q4 starts to noticeably hallucinate," "Q4 of a 1.5B model can be much worse than Q8 of the same model," and "a smaller model at Q4 will outperform a larger model at Q2." For reasoning models, weight-only "4-bit is nearly lossless, but 3-bit becomes risky" (≈1.36%/0.82% drop at 4-bit for 1.5B-class distilled Qwen). Implication: at 4GB you want a 1.5–3B model at Q4_K_M/Q5, not a 7B at Q2.

**On small code models that fit ~4GB:**
- Qwen2.5-Coder ships at 0.5B/1.5B/3B/7B/14B/32B (Apache-2.0 except the 3B, which is under the Qwen-Research license). Per the Qwen2.5-Coder Technical Report (arXiv:2409.12186), **Qwen2.5-Coder-1.5B scores ~43.3% pass@1 on HumanEval and ~50.0% on MBPP** — it does *not* beat most 7B models; its real edge is parameter efficiency (roughly ~1.2GB VRAM at Q4_K_M), which is exactly what your constraint rewards. **All these headline numbers are Python-centric**, which is itself the crux of your problem.
- These benchmarks (HumanEval, MBPP, MultiPL-E) are Python-first; MultiPL-E covers 18–22 languages but **does not include Elixir**.

**On Elixir specifically (the data-scarcity core):**
- Elixir is covered in exactly one major benchmark: **McEval** (40 languages, including Elixir AND Erlang). The Qwen blog states Qwen2.5-Coder-32B-Instruct "performs excellently across more than 40 programming languages, scoring 65.9 on McEval, with impressive performances in languages like Haskell and Racket" (and 75.2 on the companion multilingual *repair* benchmark MdEval, "ranking first among all open-source models"). Note: 65.9 is a 40-language *average*, not Elixir-specific, and is for a 32B model — exact per-language Elixir pass@1 is published only as radar-chart figures. The direction is unambiguous: Elixir (functional, low-resource) scores well below Python.
- Fine-tuning small models on Elixir is feasible with proof-of-concept precedent: DockYard published a tutorial fine-tuning code models in Elixir (Axon/Bumblebee, Fill-in-the-Middle task) on the Elixir Nx GitHub org's code. The Elixir corpus would come from Hex packages + GitHub Elixir repos (the Stack v2 contains Elixir).

**On Path A (pivot/transpile) — what's verified:**
- "Pivot language" is a real MT technique now transferred to code: PLUG (cross-lingual instruction tuning via a pivot, arXiv:2311.08711), and for *code* specifically, TransCoder-IR uses LLVM IR as a pivot. An FSE 2026 paper that embeds semantically parallel snippets across 19 languages finds **"Go exhibits as a 'lingua franca' with the highest cross-language similarity,"** proposing "centroid-based intermediary code translation."
- **MultiPL-T (Cassano et al., OOPSLA 2024, arXiv:2308.09895)** is the strongest evidence for the *spirit* of your idea: it uses StarCoderBase-15B to translate high-resource Python training data into low-resource languages (Racket, OCaml, Lua, R, Julia), keeping only translations that pass MultiPL-E-translated unit tests, then fine-tunes — achieving SOTA on those languages. Crucially, "data generated from a smaller model (StarCoderBase-15B) improves the performance of larger and better models (Code Llama 34B and 70B)." This is "use abundant Python to bootstrap a scarce language," at *training time*.
- **CodeChemist (2025, arXiv:2510.00501)** does it at *test time*: generate+execute in a high-resource PL to derive language-agnostic test oracles, then hedge-sample candidates in the low-resource PL and select by pass rate — improvements "up to 69.5%" on Lua, no retraining.
- TransCoder/TransCoder-ST established neural code translation; TransCoder-ST's lesson is that **automated unit tests are what make translation trustworthy** (+12.6 points CA@1). The recurring theme across all this work: *tests/execution feedback*, not the translation step alone, preserve correctness.

**On Path A — what's NOT there (important negative result):**
- **There is no working deterministic, rule-based Python→Elixir source-to-source transpiler** anywhere (GitHub/Hex.pm/Elixir Forum, as of June 2026). "boisgera/python-to-elixir" is just tutorial Livebook notebooks (0 stars, not a transpiler). The space is covered only by runtime interop (Pythonx, ErlPort, Pyrlang, snakebridge), a Python interpreter on the BEAM (ivarvong/pyex — a runtime interpreter, not a translator), and non-deterministic AI converters (CodeConvert, Codeporting, which explicitly warn output "is not 100% deterministic"). **Your Pylixir is genuinely novel territory.**
- The semantic gap is well-recognized: Python's mutable reassignment vs. Elixir immutability, classes → modules/structs, loops → recursion/Enum/for-comprehensions, threading → Task/GenServer/OTP. These are exactly the hard parts that make a *faithful* deterministic transpiler difficult, and they are discussed only in forum Q&A and manual-port blog posts — never in a shipping tool.

**On Path B techniques:**
- **Grammar-constrained decoding** (llama.cpp GBNF) can *guarantee* syntactically valid output by masking invalid tokens; a tree-sitter/EBNF Elixir grammar could force valid Elixir. **But** there's a documented "format tax": strict constraints "often diminish the reasoning capabilities of LLMs" (CRANE, arXiv:2502.09061; Tam et al. EMNLP 2024). The fix is to interleave free-form reasoning with a constrained *final* output. So GBNF helps concern #1 (garbage syntax) but can hurt concern #2 (logic) if applied to the whole generation.
- **Self-repair loops** (your Credence's domain): the evidence is sobering for weak models. Olausson et al., "Is Self-Repair a Silver Bullet for Code Generation?" (ICLR 2024, arXiv:2306.09896), analyzing Code Llama/GPT-3.5/GPT-4 on HumanEval and APPS, found "performance gains are often modest, vary a lot between subsets of the data, and are sometimes not present at all... self-repair is bottlenecked by the model's ability to provide feedback on its own code." More recently, "even a 7B model (Qwen2.5-Coder) showed limited or negative gains from prompt-based self-debugging on HumanEval" (arXiv:2604.10508). Fine-tuning specifically for repair (CYCLE, LeDex) unlocks it for small models. Repair works best with *execution/test feedback*, not pure self-critique.
- **Diffusion code LLMs** (Mercury Coder by Inception, DiffuCoder-7B-cpGRPO, Dream-Coder-7B) exist and are fast (Mercury runs >1000 tok/s on H100s), but are 7B+ commercial/research models with no evidence of running on 4GB Pascal and no demonstrated low-resource-language advantage — not a fit for your constraints.

## Details

### Is your intuition about logic errors being language-independent correct?
**Partly — and this distinction is the crux.** Reasoning about *what OHLC means* (open = first trade in the window, high = max, low = min, close = last; and "is this candle still forming relative to *now*?") is a specification/reasoning problem that exists identically in Python and Elixir. Generating in Python does **not** make the model understand the time-window boundary better. The self-repair and constrained-decoding literature points the same way: logic failures are reasoning failures that persist across formats/languages.

What the pivot *does* help: the model has seen vastly more correct Python time-bucketing/OHLC code (pandas `resample`, `groupby`) than correct Elixir. So the *probability of sampling a logically-correct program is higher in Python* — not because logic is language-dependent, but because the model's competence (training-data density) is. Routing through Python raises the logic floor **indirectly via competence**, while a deterministic transpile step handles syntax/idiom. This is exactly the MultiPL-T / CodeChemist insight. The catch: the transpiler must be behavior-preserving, and Python→Elixir has a wide semantic gap. For pure, functional, data-transformation logic (OHLC aggregation over a list of trades) the gap is *narrow* and a transpiler is plausible. For stateful/concurrent logic (a GenServer accumulating candles in real time, deciding window boundaries against "now") the gap is *wide* and deterministic transpilation will fight you — and incidentally, that stateful "when did the interval start accumulating" logic is precisely the kind that does NOT live cleanly in Python either.

### What is the realistic floor?
Asking a raw 4GB (1.5–3B, Q4) model to one-shot a correct, idiomatic, *stateful* GenServer is, in your words, daydreaming — corroborated by Elixir's low-resource status in McEval and the small-model self-repair evidence. But the floor moves dramatically with the right architecture:
1. **Fine-tune for Elixir** (MultiPL-T-style) → raises base competence on the target language.
2. **Generate the pure-logic core in Python** (where competence is highest), derive **test oracles** there (CodeChemist), then either transpile (Pylixir) or re-generate-and-select in Elixir against those oracles.
3. **Constrain syntax** with a GBNF Elixir grammar only on the final emission, leaving reasoning free (CRANE pattern).
4. **Close the loop with execution** — your Credence plus generated tests — because every strand of evidence says tests/execution feedback are what actually rescue both syntax and logic.

### Novel vs. replicative map
- **Replicative (do this, it's proven):** QLoRA-fine-tune a small Qwen2.5-Coder on a synthetic Elixir corpus (MultiPL-T); test-oracle-based candidate selection (CodeChemist/TransCoder-ST); GBNF-constrained final emission; execution-feedback repair.
- **Genuinely novel (your white space):** (a) A *deterministic* Python→Elixir transpiler (Pylixir) — nobody has built one; the whole space is interop or AI. (b) Restricting generation to an "Elixir-looking Python" subset designed for clean transpilation — analogous to type-directed/grammar-aligned DSL generation but unexplored for this language pair. (c) Combining a behavior-preserving AST auto-fixer (Credence) with a pivot pipeline as an integrated *local* toolchain on weak hardware.

## Recommendations

**Stage 1 — Fix the foundation (highest ROI, mostly proven):**
1. Standardize on **Qwen2.5-Coder-1.5B (Apache-2.0) at Q4_K_M/Q5_K_M GGUF**, llama.cpp CUDA build, Flash Attention OFF, modest KV cache (Q8 KV if memory-pressed). Avoid 7B-at-Q2.
2. **Build an Elixir QLoRA fine-tune using the MultiPL-T recipe**: use your existing 16–20B Qwen offline to translate documented Python functions (with their generated, validated unit tests) into Elixir, keeping only translations whose tests pass after compilation. This is the single most defensible attack on data scarcity, and MultiPL-T's "smaller generator improves larger models" result means your generator is plenty strong. Corpus seed: Hex packages + GitHub Elixir + Elixir stdlib docs.
- *Benchmark/threshold:* evaluate on a held-out Elixir McEval-style set; if the fine-tune doesn't clearly beat base Qwen-1.5B Elixir pass@1, the bottleneck is synthetic-corpus quality (test coverage), not the method — invest there before scaling model size.

**Stage 2 — Add the pivot for pure-logic kernels (novel, scoped):**
3. Use the pivot **only for stateless, functional data-transform logic** (OHLC aggregation, windowing math). Generate in Python, derive test oracles in Python (CodeChemist), then either (a) deterministically transpile via Pylixir, or (b) generate Elixir candidates and select by the Python-derived oracles. Do **not** route GenServer/OTP/stateful concurrency through Python — generate that natively in Elixir.
4. Invest Pylixir effort in a **restricted "Elixir-shaped Python" subset** (no mutation, comprehensions only, explicit returns, no classes) and prompt the model to emit that subset. This shrinks the semantic gap to something a deterministic transpiler can actually guarantee behavior-equivalence over.

**Stage 3 — Guarantees and repair:**
5. Add a **GBNF Elixir grammar** constraint to the *final* emission step only (CRANE-style: reasoning free, output constrained) to eliminate non-compiling syntax — but A/B test it, because the format tax can reduce logic pass rates.
6. Make **execution + generated tests** the spine of Credence's loop. Prefer test/compile feedback over self-critique (the self-repair literature is clear that small-model self-critique alone underperforms). Consider a small repair-specialized LoRA (CYCLE/LeDex) if prompt-based repair stalls.

**What would change these recommendations:**
- If your fine-tuned 1.5B still produces broken Elixir after a quality MultiPL-T corpus → escalate model size via CPU+GPU offload (accept slower tokens) rather than staying strictly GPU-only.
- If GBNF constraint drops logic pass@1 → switch to post-hoc parse-and-repair instead of decode-time constraint.
- If Pylixir keeps failing on real tasks → that's a signal the tasks are stateful; pull them back to native Elixir generation.

## Caveats
- **Evidence gap:** I could not obtain an exact published Elixir pass@1 number for any model — McEval reports per-language results as charts, not text. The 65.9 McEval figure is a 40-language average for Qwen2.5-Coder-32B, not Elixir-specific and not for a 4GB model. Treat "Elixir scores far below Python" as well-supported in direction but not precisely quantified.
- **MultiPL-T/CodeChemist were validated on Racket/OCaml/Lua/R/Julia, not Elixir.** Elixir's actor/OTP model is further from Python than those targets, so transfer may be weaker for stateful code. This is reasoned inference, not a measured result.
- **Diffusion LLMs and speculative decoding** were reviewed and judged out-of-scope for 4GB Pascal (model size; no demonstrated low-resource benefit). Speculative decoding aids speed, not quality, and needs VRAM for two models.
- Practitioner quantization claims (Q4/Q3/Q2 quality cliffs) come largely from high-quality community testing rather than peer review; the cited academic quantization papers support the same direction.
- The "Pascal can't do Flash Attention / bf16, FP16 is 1/64 rate on GP104" claims are verified against NVIDIA documentation and the llama.cpp issue tracker and are firm.
- "No deterministic Python→Elixir transpiler exists" reflects a thorough but not exhaustive search of GitHub, Hex.pm, and the Elixir Forum as of June 2026; a private/unpublished tool could exist, but the public ecosystem has none.
